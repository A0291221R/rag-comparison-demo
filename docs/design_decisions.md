# Design Decisions

## 1. Why LangGraph over Custom State Machines

LangGraph provides:
- **Typed state** via TypedDict — every node receives and returns a validated dict
- **Conditional edges** with clean Python functions — no ad-hoc if/else chains
- **Built-in checkpointing** via MemorySaver or Postgres — session state persists across requests
- **`recursion_limit`** enforcement — prevents infinite agent loops without custom logic
- **`interrupt_before`** — human-in-the-loop support at any node boundary
- **Native async** — all nodes are `async def`, compatible with FastAPI and asyncio.gather

Alternative considered: raw LangChain LCEL chains — rejected because they lack first-class
conditional branching, persistent state, and loop control needed for self-reflection.

## 2. Agentic RAG vs GraphRAG — When to Use Which

### Agentic RAG strengths
- Open-domain queries with no clear named entities
- Summarization over heterogeneous documents
- Queries requiring multi-step reasoning over unstructured text
- When knowledge graph construction is not feasible (corpus too dynamic)

### GraphRAG strengths
- Entity-centric queries: "How are X and Y related?"
- Multi-hop relationship traversal: "Who co-authored with X who also studied Y?"
- Queries requiring structured provenance (precise citation chains)
- Domains with stable, well-defined ontologies (biomedical, legal, financial)

### Auto-routing logic
The orchestrator classifies each query using a cheap model (gpt-4o-mini) before routing.
Heuristics: presence of multiple proper nouns, relational verbs ("related to", "connected"),
and comparison operators signal GraphRAG. Open-ended "explain" or "summarize" signal Agentic.

## 3. Hybrid Search: RRF Fusion over Learned Sparse

**Choice**: Dense (OpenAI embeddings) + Sparse (BM25) with Reciprocal Rank Fusion.

**Why not learned sparse (SPLADE, uniCOIL)?**
- Requires additional model inference per query (latency cost)
- BM25 is sufficient for exact keyword matching in most RAG corpora
- SPLADE adds ~30% latency; RRF captures most of the recall benefit

**Why not pure dense?**
- Dense models miss exact-match queries for rare terms, product names, model versions
- Hybrid consistently outperforms dense-only on BEIR benchmarks by 3–8% Recall@10

**RRF formula**: `score(d) = Σ 1 / (k + rank_i(d))` where k=60 (Qdrant default)

## 4. Neo4j 5.x Vector Index vs Dedicated Vector Store

**Choice**: Qdrant as primary vector store; Neo4j vector index for graph-local similarity.

**Why not Neo4j-only?**
- Neo4j vector search lacks HNSW parameter tuning depth of Qdrant
- Qdrant has better horizontal scaling and snapshot/backup tooling
- Neo4j vector index is used only for within-graph nearest-neighbor (e.g., "find chunks
  semantically similar to this entity's description")

**Why not Weaviate?**
- Qdrant has simpler API, better async Python client, and native sparse vector support
- Weaviate's schema system adds overhead for this demo's dynamic corpus

## 5. Token Budget Enforcement

**Strategy**: Hard cap enforced at the orchestrator level before each LLM call.

```python
if state["token_usage"]["total"] >= settings.token_budget_per_query:
    # skip remaining LLM calls, return best answer so far
```

Additionally:
- **Classification/grading nodes** use `gpt-4o-mini` (~10× cheaper than gpt-4o)
- **Generation node** uses `gpt-4o` only for final answer synthesis
- **Prompt compression** (LLMLingua) is available as an optional step for long contexts

This model-routing pattern typically reduces cost by 40–60% vs using gpt-4o throughout.

## 6. Embedding Fine-Tuning Pipeline

**Approach**: Contrastive learning with TripletLoss on (anchor, positive, negative) triples.

- **Anchors**: Synthetic questions generated by gpt-4o-mini over corpus chunks
- **Positives**: The source chunk that answers the question
- **Negatives**: A chunk from a different document (hard negative mining)

**Why TripletLoss over MultipleNegativesRankingLoss (MNRL)?**
- MNRL is faster and often better, but requires in-batch negatives (large batch sizes)
- TripletLoss works well with smaller datasets and explicit hard negatives
- For production: upgrade to MNRL with a mined hard-negative dataset

**Version tagging**: Every fine-tuned model writes a `version_manifest.json` with
training metadata. The `.env` `EMBEDDING_MODEL_FT` flag enables instant rollback.

## 7. Observability Stack Selection

| Tool | Role | Why |
|------|------|-----|
| **LangSmith** | LangGraph-native trace capture | First-class LangGraph integration |
| **OpenTelemetry** | Vendor-neutral spans/metrics | Portable; works with Jaeger, Grafana Tempo |
| **Jaeger** | Distributed trace UI | Free, easy Docker setup |
| **Prometheus** | Metrics scraping | Industry standard; Grafana native |
| **structlog** | Structured JSON logs | Better than `logging` for correlation IDs |

All trace IDs propagate through: LangGraph state → FastAPI response headers →
Prometheus labels → LangSmith runs. This enables full cross-system correlation.

---

## 8. gRPC for Streaming Generation

The gRPC endpoint supports token-by-token streaming via server-side streaming RPCs.
This enables:
- Lower time-to-first-token for UI clients
- Progress indication during long graph traversals
- Backpressure-aware clients

The REST API returns complete responses (no streaming) for simplicity and compatibility
with standard HTTP clients and the Gradio UI.
